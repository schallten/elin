v0.4.0

Libraries & Tooling — Ecosystem

Items 12–21 ship a standard library and developer toolchain, alongside the core bump allocator (M1) for dynamic allocation. This is how we turn ELIN from a bare VM into a usable programming environment.

Libraries // Item 12 ✓ Done

Standard I/O library

Three opcodes for interactive programs: READ (69) reads a line from stdin into the string pool and pushes the handle; WRITE (70) prints a string handle without newline; FLUSH (71) flushes stdout. read(), write, and flush keyword syntax in the compiler.

Libraries // Item 13 ✓ Done

String operations library

Four opcodes for text handling: STRLEN (72) pushes the length of a string handle; STRCAT (73) concatenates two handles and pushes a new one; SUBSTR (74) extracts a substring by handle, offset, and length; STRCMP (75) compares two handles and pushes -1/0/1. Accessed via strlen(), strcat(), substr(), and strcmp() in source.

Libraries // Item 14

File I/O library

Four opcodes: FOPEN (76), FREAD (77), FWRITE (78), FCLOSE (79). On PC they wrap the C stdio API; on ESP they wrap LittleFS. Same API surface, different backend. File handles are integer descriptors into a fixed-size table we maintain inside the VM.

Libraries // Item 15

Time library

We need four opcodes for anything real-time: TIME (80) pushes milliseconds since boot (std::chrono on PC, millis() on ESP); DELAY (81) blocks for N ms; RTC_READ (82) and RTC_WRITE (83) access ESP RTC memory (no-op on PC). Every nontrivial program we write needs at least TIME and DELAY.

Libraries // Item 16

Random number library

Two opcodes: RAND (84) pushes a random 64-bit integer; SRAND (85) seeds the generator. On PC we use std::mt19937_64; on ESP we use the hardware random(). Same interface, platform-appropriate implementation underneath.

Libraries // Item 17

FFI (CALL_EXTERN)

This is how we bridge ELIN to the host system. A source declaration like EXTERN "math" sin; emits CALL_EXTERN (86). We maintain a function registry per platform: C++ stdlib on PC, GPIO and WiFi on ESP. Same opcode, different registration tables.

Tooling // Item 18

Bytecode assembler

We're building elin-asm — a human-readable text format that assembles to .outz:

PUSH 42
STORE 0
HALT

We want to hand-write optimized bytecode, test individual opcodes, and bootstrap the compiler — all without hex-editing raw .outz files.

Tooling // Item 19

Source-level debugger

We're building elin-debug: set breakpoints, single-step through source lines, inspect state. The compiler will emit a line-number table in .outz that maps source lines to bytecode offsets. On PC it's an interactive terminal; on ESP it accepts serial commands (s = step, c = continue, p = print stack).

Tooling // Item 20

REPL

We're building elin-repl: read a line of ELIN source, compile it, run it, print the result, repeat. This requires making the compiler embeddable as a library (or building a separate JIT mode). On PC it gives us immediate feedback for learning; on ESP it lets us write interactive shell commands in ELIN instead of hardcoded C++.

Tooling // Item 21

Package manager

We're building elin-pkg to manage dependencies:

  • elin-pkg init — creates an elin.json manifest
  • elin-pkg add math — pulls from a git repo or local path
  • elin-pkg build — compiles everything together

Dependencies are just .elin files with their own elin.json. No registry server — git URLs or local paths. This also serves as a prototype for Loom so we can figure out what it really needs before writing it in C++ for ESP.

Memory // Item M1

Core bump allocator

A single heap segment using bump-pointer allocation. Each allocation returns a handle (integer ID) rather than a raw address. A flat handle table maps IDs to block metadata (pointer, size, validity flag). Allocation is O(1) — increment a pointer. Deallocation is explicit: FREE invalidates the handle and recycles its slot; memory is not reclaimed, the bump pointer only advances. LOAD_H / STORE_H opcodes access cells with bounds checking. Use-after-free, double-free, and OOB are caught at runtime — no silent corruption.

Compiler // Item —

Port the compiler to Gleam

The Python compiler was deliberately written without classes — just pure functions taking data in and returning data out — so it could be ported to another language. We're moving it to Gleam, a type-safe functional language with algebraic data types and exhaustive pattern matching.

This isn't just cosmetic. Our 17 @dataclass AST nodes become a single type Expr with variants. Every isinstance check in Python becomes a compile-time exhaustiveness check in Gleam — if we add a new AST node and forget to handle it in a case, the compiler refuses to build. Our manual state threading (state = generate(x, state); state = generate(y, state)) collapses into clean pipelines (state |> generate(x) |> generate(y)). The whole codebase shrinks by roughly 40%.

More importantly, a cleaner compiler is easier to port to ELIN itself for v0.5.0. If the Python compiler was already a stepping stone to self-hosting, the Gleam version is a sturdier one — fewer bugs, smaller surface area, and a type system that catches mismatches before runtime.

v0.5.0

Memory + Self-Hosting — The Turning Point

Multi-segment memory regions (M2) provide temp allocation for function-local work without manual FREE, powering the self-hosting compiler (Item 22). Once ELIN can compile itself, it stops being a toy and becomes a self-sustaining system.

Memory // Item M2

Multi-segment architecture + regions

Four logical bump-allocated arenas: Main (0) for explicitly managed allocations, Temp (1) for function-local scratch (compiler auto-inserts REGION_ENTER/REGION_EXIT around function bodies), Interrupt (2) for ISR-safe allocations, and Spare (3) for overflow or special-purpose use. REGION_EXIT resets the bump pointer and invalidates all handles in that segment — no per-object tracking needed. Recursive functions and loops can allocate temporaries without leaking and without manual FREE.

Self-Hosting // Item 22

Write the ELIN compiler in ELIN

Our endgame: write the ELIN compiler in ELIN itself. That's the proof that we've built something complete. We'll break it into three stages:

Stage 1: Write a lexer in ELIN. It takes a string handle to source code and produces an array of token structs (type, value). We'll validate it against the output of our Python/Haskell lexer.

Stage 2: Write a parser in ELIN. It consumes the token array and produces an AST using nested arrays or structs.

Stage 3: Write a bytecode emitter in ELIN. It walks the AST and writes valid .outz bytes through the file I/O library (Item 14).

Prerequisite: we need structs or records first — or a convention of using arrays with fixed offsets:

struct Token {
    int type;
    str value;
};

We'll start with Stage 1 only. The moment our ELIN lexer produces the same token stream as the reference implementation for a real program, we'll know the language is viable for self-hosting.

v0.6.0

Memory — Hardening & PC Comfort

Embedded-hardening (M3) makes memory usage fully predictable at link time, while PC features (M4) add dynamic growth, compaction, and profiling for long-running desktop/server programs.

Memory // Item M3

Embedded hardening

Segments bind to statically provided memory blocks (e.g. static ll main_heap[2048]) — no malloc or C++ heap usage. The VM performs zero runtime allocation; all memory is accounted for in the linker map. ALLOC returns null (handle 0) on segment full — no exceptions, no crashes. Generation counters make temp reclamation O(1): each REGION_EXIT increments a per-segment generation number, and handle validity becomes a generation comparison instead of a table scan. SEG_USED opcode reports current utilization for runtime pressure monitoring. Published worst-case cycle counts for all memory ops enable hard real-time scheduling.

Memory // Item M4

PC comfort features

Additive features gated behind build flags: dynamic growth allows segments backed by malloc/new to grow on demand; compaction moves live objects to segment start and can run incrementally across VM cycles to avoid visible pauses; profiling hooks provide runtime callbacks for allocation events and segment pressure thresholds. All PC features are optional — the embedded binary remains unchanged.
