Roadmap

Where We're Taking ELIN

Our wishlist. No deadlines. Subject to whims.

v0.3.0

Core VM — Foundation

Our first eleven items harden the instruction set and bytecode format. By the end of this release we want a VM that is capable, efficient, and pleasant to target.

Core VM // Item 01

PRINT pops from eval_stack

Right now PRINT reads from a global slot, forcing the compiler to emit a STORE/LOAD round-trip just to print a value. We're changing PRINT (8) to pop the top of eval_stack and print it directly. Then print r; becomes simply LOAD 2; PRINT; — no more temporary juggling. Same for PRINT_STR.

Core VM // Item 02

Constant pool + PUSH_CONST

Every PUSH costs us 5 bytes (1 0 0 0 value). For repeated constants — zeros, ones, common values — that adds up fast. We're adding a constant pool to the .outz format and a PUSH_CONST <idx> opcode that references pool entries with a single byte. Same expressiveness, much denser bytecode.

Core VM // Item 03

DUP, DROP, SWAP

We're reserving IDs 60-62 for the three fundamental stack operations every stack VM needs: DUP duplicates the top, DROP discards it, SWAP exchanges the top two. Without them we end up with STORE/LOAD hacks — more bytes, more globals, more noise.

Core VM // Item 04

NEG and NOT

Today -x compiles to PUSH 0; LOAD x; SUB; — three instructions for one operation. Logical not (!x) has no dedicated opcode at all. We're adding NEG (63) and NOT (64). NEG pops a and pushes -a. NOT pushes 1 if a == 0 and 0 otherwise — natural for our boolean-as-int model.

Core VM // Item 05

NOP

We're adding a no-operation instruction at ID 0 or 65. It advances the program counter and does nothing else — useful for padding, alignment, and placeholder slots during optimization.

Core VM // Item 06

INC and DEC

Increment and decrement are the most common operations in almost every program. Right now i = i + 1 compiles to LOAD 5; PUSH 1; ADD; STORE 5; — 4 instructions, 11 bytes. With IDs 66-67, INC 5 does the same in 2 bytes. A cheap win for a pattern we use constantly.

Core VM // Item 07

DEBUG / TRACE mode

We have zero visibility into VM execution today. Compiler bugs look the same as runtime bugs and we have no way to tell them apart. We're building a flag that prints every opcode, eval_stack state, and frame info — so we can finally see what's happening inside.

Core VM // Item 08

INPUT opcode

ELIN is output-only right now. We're adding INPUT (68): it blocks until a number arrives (serial on ESP, stdin on PC), then pushes it onto eval_stack. This unlocks everything interactive — calculators, prompts, games.

Core VM // Item 09

MOD and ABS

We need MOD for array indexing, circular buffers, and clock arithmetic. We need ABS for distance calculations. IDs 55-56 slot right into our 64-bit integer model and we reach for both constantly in embedded code.

Core VM // Item 10

Version the .outz format

Our bytecode format has no header — raw opcodes from offset zero. We're fixing that with magic bytes (0x45 0x4C 0x49 0x4E = "ELIN"), a version byte, flags, and pool offsets. This lets us evolve the format over time, detect corruption, and reject unsupported files gracefully.

Core VM // Item 11

Compile-time constant evaluation

If every operand in an expression is a compile-time constant, the compiler folds the result directly into the bytecode instead of emitting runtime instructions. 42 + 15 becomes a single PUSH_CONST 57 rather than PUSH 42; PUSH 15; ADD; — the work happens once, at compile time, and the runtime never sees the arithmetic. The same applies to any pure operation — unary, binary, logical — whose inputs are all known at compile time. The interpreter never executes what the compiler already knows.

We'll need to be careful not to over-fold. A function that never calls an impure opcode (WRITE, FOPEN, CALL_EXTERN, etc.) is pure by default — but tracing the call graph to prove that won't make it into the first pass. For now we'll only fold literals and expressions built from them: no function calls, no variables (even const, unless we can trace the value locally), and no array accesses (unless the array is const and the index resolves at compile time).

v0.4.0

Libraries & Tooling — Ecosystem

Items 12–21 are where we ship a standard library and developer toolchain. This is how we turn ELIN from a bare VM into a usable programming environment.

Libraries // Item 12

Standard I/O library

We're building three opcodes for interactive programs: READ (69) reads a line from stdin into a string pool slot and pushes the handle; WRITE (70) takes a handle and prints without a newline; FLUSH (71) forces pending output. Prompts, menus, dialogs — all compile identically for PC and ESP (where READ draws from a serial buffer).

Libraries // Item 13

String operations library

Four opcodes we need for real text handling: STRLEN (72), STRCAT (73), SUBSTR (74), STRCMP (75). They operate on string pool handles passed as 64-bit integers. On PC we delegate to C++; on ESP we use pre-allocated buffers. No dynamic allocation — STRCAT fails cleanly if the result exceeds our fixed max.

Libraries // Item 14

File I/O library

Four opcodes: FOPEN (76), FREAD (77), FWRITE (78), FCLOSE (79). On PC they wrap the C stdio API; on ESP they wrap LittleFS. Same API surface, different backend. File handles are integer descriptors into a fixed-size table we maintain inside the VM.

Libraries // Item 15

Time library

We need four opcodes for anything real-time: TIME (80) pushes milliseconds since boot (std::chrono on PC, millis() on ESP); DELAY (81) blocks for N ms; RTC_READ (82) and RTC_WRITE (83) access ESP RTC memory (no-op on PC). Every nontrivial program we write needs at least TIME and DELAY.

Libraries // Item 16

Random number library

Two opcodes: RAND (84) pushes a random 64-bit integer; SRAND (85) seeds the generator. On PC we use std::mt19937_64; on ESP we use the hardware random(). Same interface, platform-appropriate implementation underneath.

Libraries // Item 17

FFI (CALL_EXTERN)

This is how we bridge ELIN to the host system. A source declaration like EXTERN "math" sin; emits CALL_EXTERN (86). We maintain a function registry per platform: C++ stdlib on PC, GPIO and WiFi on ESP. Same opcode, different registration tables.

Tooling // Item 18

Bytecode assembler

We're building elin-asm — a human-readable text format that assembles to .outz:

PUSH 42
STORE 0
HALT

We want to hand-write optimized bytecode, test individual opcodes, and bootstrap the compiler — all without hex-editing raw .outz files.

Tooling // Item 19

Source-level debugger

We're building elin-debug: set breakpoints, single-step through source lines, inspect state. The compiler will emit a line-number table in .outz that maps source lines to bytecode offsets. On PC it's an interactive terminal; on ESP it accepts serial commands (s = step, c = continue, p = print stack).

Tooling // Item 20

REPL

We're building elin-repl: read a line of ELIN source, compile it, run it, print the result, repeat. This requires making the compiler embeddable as a library (or building a separate JIT mode). On PC it gives us immediate feedback for learning; on ESP it lets us write interactive shell commands in ELIN instead of hardcoded C++.

Tooling // Item 21

Package manager

We're building elin-pkg to manage dependencies:

  • elin-pkg init — creates an elin.json manifest
  • elin-pkg add math — pulls from a git repo or local path
  • elin-pkg build — compiles everything together

Dependencies are just .elin files with their own elin.json. No registry server — git URLs or local paths. This also serves as a prototype for Loom so we can figure out what it really needs before writing it in C++ for ESP.

v0.5.0

Self-Hosting — The Turning Point

Item 22 is the whole release. Once ELIN can compile itself, it stops being a toy and becomes a self-sustaining system. This is the milestone everything else is building toward.

Self-Hosting // Item 22

Write the ELIN compiler in ELIN

Our endgame: write the ELIN compiler in ELIN itself. That's the proof that we've built something complete. We'll break it into three stages:

Stage 1: Write a lexer in ELIN. It takes a string handle to source code and produces an array of token structs (type, value). We'll validate it against the output of our Python/Haskell lexer.

Stage 2: Write a parser in ELIN. It consumes the token array and produces an AST using nested arrays or structs.

Stage 3: Write a bytecode emitter in ELIN. It walks the AST and writes valid .outz bytes through the file I/O library (Item 14).

Prerequisite: we need structs or records first — or a convention of using arrays with fixed offsets:

struct Token {
    int type;
    str value;
};

We'll start with Stage 1 only. The moment our ELIN lexer produces the same token stream as the reference implementation for a real program, we'll know the language is viable for self-hosting.
